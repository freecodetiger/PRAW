# 为什么项目前后端选择 Tauri + React + Rust

## 结论
这套方案适合这个项目，不是因为它“新”，而是因为它刚好把三个本质问题拆开了：

- React 负责高频 UI 状态与交互表达
- Rust 负责终端会话、系统能力、性能敏感逻辑
- Tauri 负责在桌面环境里把两者桥接起来

这个边界符合项目的第一性原理：
智能终端首先是“桌面终端”，其次才是“带 AI 的应用”。

## 这个项目真正需要的能力
从需求上看，这个项目不是一个普通 Web 页面，而是一个需要长期演进的桌面工作区。它至少需要：

- 在 Ubuntu 上稳定启动和分发
- 管理真实 shell / PTY 会话
- 持续接收高频终端输出
- 支持复杂窗口布局、分屏、拖拽、历史转录
- 做本地配置、窗口快照、工作区恢复
- 接入 AI 能力，但 AI 只是增强层，不应反客为主

这类需求天然分成两部分：

- 系统侧能力
- 交互侧能力

因此前后端必须明确分层，而不能把所有逻辑塞进前端。

## 为什么是 Tauri
Tauri 在这里的价值不是“能做桌面应用”这么简单，而是它让项目获得了三个关键特性。

### 1. 桌面能力是原生的，不是伪装出来的
终端、窗口快照、事件桥接、文件存储，这些都不是浏览器原生擅长的事情。Tauri 允许前端通过命令调用 Rust 后端，把系统能力稳定地暴露出来。

这让项目可以自然地拥有：

- 终端会话创建与销毁
- 输出事件监听
- 本地配置读写
- AI 请求与本地补全桥接

### 2. 保留 Web UI 开发效率
这个项目的 UI 不是静态页面，而是高交互工作区：

- 分屏布局
- 输入框与 transcript 双模式
- 设置面板
- 右键菜单
- AI mode 主题与状态切换

这类高频状态变化，用 React 实现明显比纯 Rust UI 工具链更高效，也更容易迭代。

### 3. 比 Electron 更克制
这个项目本质上是终端，不应该为了一层 UI 壳付出过大的运行时代价。Tauri 的定位更符合“原生能力 + 轻量 WebView”的思路。

对于终端这类需要长期常驻的桌面程序，克制的资源模型比过度豪华的壳更重要。

## 为什么前端用 React
React 在这个项目中承担的不是“页面展示”，而是复杂交互状态机的承载层。

当前前端已经在承担这些职责：

- 窗口布局树呈现
- pane 激活态与聚焦切换
- dialog / classic / AI mode 呈现切换
- transcript 渲染
- ghost completion 交互
- 设置项编辑与主题选择

这些能力的共同特点是：

- 状态变化频繁
- 组件层级较深
- 需要显式数据流
- 需要快速局部重渲染

React 很适合做这一层，因为它擅长把“当前状态”稳定地投影为“当前界面”。

## 为什么后端用 Rust
Rust 在这个项目里不是为了“炫技”，而是因为后端承担的是最不适合随意出错的部分。

当前 Rust 侧已经负责或适合继续负责：

- 终端会话管理
- PTY / shell 集成
- 输出事件派发
- 配置持久化
- 工作区快照存储
- AI 请求桥接
- 本地补全能力

这些能力的共同特点是：

- 生命周期长
- 和系统资源直接交互
- 对稳定性和错误边界敏感
- 需要明确控制并发和所有权

对于终端程序来说，后端一旦出错，代价通常比前端 bug 更大，因为它直接影响会话、进程、输出流和数据可靠性。Rust 在这一层的收益是实打实的。

## 为什么不是“前端全做”
如果把终端、AI 请求、配置存储、工作区恢复都放在前端，一开始会显得快，但后面会遇到几个明显问题：

- 系统能力边界不清晰
- 输出流处理越来越难维护
- 状态与副作用耦合在一起
- 调试时很难分辨是 UI 问题还是运行时问题

这个项目后续还会继续增长，前端必须保持“负责交互，不直接承担系统运行时”的边界，否则复杂度会快速失控。

## 为什么不是“后端全做 UI”
如果把界面也转到纯原生 UI 或 Rust UI 框架，理论上可以更统一，但不适合当前阶段，原因很现实：

- UI 迭代速度会明显下降
- 复杂分屏与交互试错成本更高
- 样式、主题、设置面板之类的调整会变重
- AI mode、dialog transcript 这类体验实验会变慢

这个项目还在快速形成交互范式，UI 层需要足够灵活，React 正好提供了这种灵活性。

## 当前架构的合理边界
这套方案最有价值的地方，不是技术栈名字本身，而是职责边界比较清楚。

### 前端
负责：

- 视图状态
- 布局求解结果的呈现
- 输入交互
- 模式切换
- 主题与设置 UI

不负责：

- PTY 生命周期
- 真实 shell 读写
- 本地持久化细节
- 底层 AI 网络调用细节

### 后端
负责：

- 终端会话与系统资源
- 事件发射
- 存储
- AI / completion 能力的实际执行

不负责：

- 复杂 UI 状态管理
- pane 呈现策略
- 前端主题与交互细节

### Tauri 桥
负责：

- 命令调用
- 事件订阅
- 前后端协议

它是边界层，不是业务中心。

## 为什么这套方案适合继续演进
这个项目后续还会继续增加：

- 更强的幽灵补全
- 更成熟的 AI mode
- 更丰富的主题与设置
- 更稳定的工作区恢复
- 更多终端行为兼容

如果底层栈选型本身就是耦合的，每加一个功能都要跨层重写。当前这套 `Tauri + React + Rust` 的价值就在于：

- UI 可以快速试验
- 系统能力可以稳定沉淀
- 前后端契约可以逐步变清晰
- 后续做分发也顺手

## 一句话总结
这套方案不是折中，而是分工明确：

- React 让交互复杂度可控
- Rust 让运行时复杂度可控
- Tauri 让桌面能力与 Web UI 之间的连接可控

对于一个真正要长期使用的 Ubuntu 智能终端来说，这种“边界清楚的组合”比单纯追求某一侧统一更重要。
