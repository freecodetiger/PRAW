# 为什么从二叉树改成多叉容器树

## 背景
这个项目的终端工作区不是简单的左右分栏，而是要求同时支持：

- 一个窗口内自由分屏
- 多次连续上下或左右切分
- 拖拽重排 pane
- 边界拖拽时始终保证每个 pane 的标题区、内容区、输入区完整可见
- 靠近窗口边界的 pane 像“撞墙”一样不能被继续压缩

如果继续使用严格二叉分割树，这些目标会越来越依赖补丁式逻辑，最终导致局部修复互相干扰。

## 二叉树方案的问题
严格二叉树的特点是：每次 split 都生成一个父节点和两个子节点。它的优点是实现早期简单，但在这个项目里有几个结构性问题。

### 1. 同轴连续分屏会退化成嵌套链
例如右侧连续做三次上下分屏，用户视觉上看到的是三个竖向相邻 pane，但在二叉树里通常会变成“上层 split 的下半部分，再被继续 split”。

这会带来一个直接后果：
用户拖动看起来相邻的边界，实际修改的却可能是不同层级节点的 ratio。

结果就是：

- 上面的拖拽会间接影响下面的 pane
- 下层 pane 会被非预期压缩
- 贴近窗口边界的 pane 容易出现输入框被挤出可视区域的情况

### 2. 视觉相邻关系和结构相邻关系不一致
用户操作的是屏幕上的“边”，但二叉树更关心“父子关系”。

这在分屏较深时会出现错位：

- 屏幕上两个 pane 明明相邻
- 但它们不一定共享同一个直接父节点
- resize 时很难稳定地找到“这条边真正归谁管”

所以旧方案里会反复出现一种现象：
拖动一条边，影响的不是这一对 pane，而是某个上层子树。

### 3. 边界约束难以下沉到正确层级
这个项目的下边界不是普通边界。靠近窗口底部的 pane 需要为：

- 标题栏
- 内容区
- 输入区

预留完整空间。

在二叉树里，这类约束通常只能作为 ratio 的额外 clamp 叠加在局部节点上，但局部 clamp 不等于全局几何约束。只要树的嵌套关系继续变深，就会不断出现“这里修好了，另一处又穿了”的问题。

## 多叉容器树解决了什么
现在的布局模型仍然是树，但不是“严格二叉分割树”，而是“多叉容器树”。

核心节点只有两类：

- `pane`：叶子节点，对应一个实际 tab
- `container`：容器节点，包含 `axis`、`children`、`sizes`

这意味着一个纵向容器可以直接拥有 3 个、4 个甚至更多同轴子节点，而不是被迫嵌套成链。

## 关键收益

### 1. 同轴分组可以直接表达
如果右侧有三个纵向 pane，现在可以直接表示为：

- 一个 `vertical container`
- 三个子 pane
- 两条 divider

这比“右边继续套右边”更接近用户心智，也更接近真实屏幕关系。

### 2. 结构拓扑和几何求解解耦
现在的树只描述“谁和谁属于同一组、分割轴是什么”，不直接承担像素级布局的全部责任。

真正的矩形位置由独立几何求解器统一计算：

- 先根据容器树解出每个 pane 的 `rect`
- 再计算每个 pane 是否贴着窗口上下左右边界
- 最后根据真实几何结果做 resize 与约束判断

这意味着：
布局结构负责表达关系，几何求解器负责表达现实。

### 3. resize 目标从“节点 ratio”变成“真实边界”
现在拖动边界时，不再先问“改哪个父节点的 ratio”，而是先问：

- 当前 pane 的左、右、上、下边缘分别对应哪条真实 divider
- 这条 divider 的两侧最小可压缩空间是多少

这样做之后，拖拽行为会稳定很多，因为操作对象终于和用户视觉对象一致了。

### 4. 最小尺寸约束可以按 pane 位置动态计算
现在的最小高度不是静态常量，而是和 pane 所处位置有关。

例如：

- 中间的内部 pane 可以允许更小的最小高度
- 贴近窗口下边界的 pane 需要保留更大的最小高度，确保输入区完整

这正是“底部 pane 像撞墙一样不能继续下压”的基础。

### 5. 同轴容器自动拍平，避免无意义嵌套
当前布局树在分裂和拖拽后会做 normalize：

- 如果父容器和子容器轴一致，就会自动拍平
- 保留同轴 children 与 sizes
- 去掉没有语义价值的中间层

这点非常重要，因为它避免了布局树在长期使用中不断长成深链，减少了历史操作对当前结构的污染。

## 为什么这更适合终端工作区
终端工作区和普通 IDE 面板不完全一样。它更强调：

- pane 完整性
- 高频拆分与关闭
- 连续分屏后的稳定性
- 输入区、标题区的强约束
- 运行时拖拽和重排

多叉容器树不是为了“理论上更高级”，而是因为它更接近这个项目的真实操作模型。

简单说：

- 二叉树更适合描述“逐次切割历史”
- 多叉容器树更适合描述“当前屏幕布局事实”

这个项目更需要后者。

## 一句话总结
从二叉树迁移到多叉容器树，本质上不是换了一个数据结构名字，而是把布局系统从“围绕 split 历史补逻辑”，改成了“围绕当前几何事实求解和约束”。

这让分屏、拖拽、边界限制、最小尺寸保护这些需求第一次可以在同一套模型里稳定成立。
