# PRAW UI 设计技术亮点

## 结论
PRAW 的 UI 不是在做一个“像 IDE 的终端壳”，而是在尝试建立一套更接近未来工作流终端的界面语言。

它当前最有价值的设计，不是某一个颜色或某一块面板，而是把几个高冲突目标同时放进了一套可维护结构里：

- 极简主界面
- 多分屏工作区
- dialog / classic 双交互模型
- AI mode 强视觉反馈
- 可切换主题
- 终端 ANSI 高亮与应用主题并存

这套体系的关键不是视觉炫技，而是“界面状态与工作状态一致”。

## 这个终端的 UI 本质上在解决什么
普通终端的 UI 其实非常单一：

- 一块输出区
- 一个滚动缓冲区
- 一个光标

但 PRAW 的目标不是复刻传统终端，而是承载更复杂的工作流：

- 多标签页
- 多分屏
- 对话式输入
- 传统终端模式回退
- AI CLI 进入后的工作模式切换
- 用户可自定义的主题系统

所以真正的设计问题不是“做得好不好看”，而是：

- 如何在极简和多功能之间不崩坏
- 如何让状态切换被用户立刻感知
- 如何让高频使用下的视觉负担足够低

## 当前 UI 架构最值得肯定的地方

### 1. 把窗口、标签页、分屏拆成不同层级
这个项目已经逐渐明确了一个很关键的层级关系：

- 窗口是第一层工作区
- 标签页是工作单元
- 分屏是标签页内部的空间分配方式

这让 UI 不再是“很多块随便拼起来”，而是有清楚的信息架构。

当用户分屏越来越多时，这种层级关系会直接影响认知负担。

### 2. 从复杂装饰退回极简画布，是正确方向
你后面持续推动的方向很明确：

- 去掉多余提示文案
- 去掉冗余圆角
- 去掉多余背景装饰
- 保持纯净大终端画布
- 只保留必要输入框和边界

这不是单纯“做减法”，而是在把界面资源让给真正重要的东西：

- 命令
- 输出
- pane 关系
- 当前模式状态

终端类产品如果 UI 存在过多装饰，用户长期使用时会持续被干扰。现在的收敛方向是对的。

### 3. dialog 模式和 classic 模式共存，但不互相伪装
这点非常关键。

很多产品会试图把所有输入范式统一成一种样子，结果两头都不满意。

PRAW 选择的是：

- 平时允许使用 dialog 模式提升可读性和工作流感
- 需要完全兼容 shell 行为时回到 classic 模式
- 进入 AI CLI 时再切到独立的 AI mode 视觉态

这说明 UI 不是静态皮肤，而是在表达运行时语义。

## AI mode 为什么是一个亮点

### 1. AI mode 本质上是“工作状态可视化”
进入 `codex`、`claude` 这类 AI CLI 后，用户面对的已经不是普通 shell 任务，而是另一种工作流：

- 多轮交互
- 长时等待
- 模型输出流
- 传统 transcript 不再是主角

如果 UI 不显式切换，用户会很容易把它误认为普通命令执行。

因此 AI mode 的价值在于：

- 让用户立即知道自己已经进入另一种工作语境
- 把普通终端和 AI 工作流隔离开
- 给用户更强的可控感

### 2. 强调边框而不是大面积重绘，是克制的设计
你最终保留的左侧色条和模式提示，本质上是很好的选择。

原因很简单：

- 状态足够显眼
- 不会遮挡内容
- 不会造成持续视觉疲劳
- 不需要重做整套布局

相比用大面积纹理、斜线、重背景去强调模式，这种方式更适合终端。

终端是长时间盯着看的工具，炫酷必须服从可读性。

## 主题系统为什么值得保留和继续扩展

### 1. 主题不是“换颜色”，而是状态 token 系统
真正可维护的主题系统，核心不是写几套 CSS，而是把 UI 的颜色语义抽象成 token。

例如：

- 背景色
- 前景文字
- 边框色
- 激活态色
- AI mode 强调色
- 输入框色
- 面板分隔色

这样做好处很直接：

- light / dark / expressive 主题共享同一套组件结构
- 用户自定义颜色时不需要改组件逻辑
- AI mode 可以基于主题 token 再叠加特殊状态

### 2. dark 主题被拉回“真正的终端黑”，方向正确
用户已经明确提出一个很好的判断标准：
dark 主题应该像传统黑底终端，而不是“偏灰的暗色 Web 页面”。

这背后反映的是两类暗色主题的本质区别：

- Web 暗色主题强调舒适面板感
- 终端暗色主题强调高对比、低干扰、强信息密度

PRAW 作为终端，更应该靠近后者。

这就是为什么需要：

- 更黑的背景
- 更亮的高亮色
- 更强的 ANSI 颜色对比
- 更克制的边框和分隔

### 3. 允许用户自定义主题色，是对工作流差异的尊重
不同用户对终端颜色偏好差异极大：

- 有人喜欢纯白极简
- 有人喜欢传统黑底
- 有人希望 AI mode 更强烈
- 有人希望整套界面更克制

允许用户自定义主题色和背景，本质上是在承认：
终端不是内容消费产品，而是生产工具。

生产工具应该允许用户把界面调成最适合自己长期工作的状态。

## 字体与高亮为什么是这套 UI 的基础设施

### 1. Nerd Font 不是装饰，而是终端可读性组件
项目切换到 `CaskaydiaCove Nerd Font` 不是审美偏好而已。

它的实际价值在于：

- 等宽表现稳定
- 代码与命令可读性高
- Nerd Font 图标兼容性好
- 在终端场景下字形完整

终端里的字体选择直接影响：

- prompt 边界清晰度
- 中英文混排稳定性
- 路径、参数、符号的辨识度
- 分屏高密度场景下的扫描速度

### 2. ANSI 高亮与应用主题必须并存，而不是互相覆盖
终端内容本身自带颜色语义，例如：

- 错误红色
- 成功绿色
- 路径蓝色
- Git / CLI 工具的语义高亮

如果应用主题把这些信息冲掉，终端就失去了最重要的即时反馈能力。

因此正确方向不是“主题统一所有颜色”，而是：

- 应用主题负责容器和界面层
- ANSI 高亮负责命令内容层

这是两个不同语义层，不能混为一谈。

### 3. dialog 历史记录高亮的价值在于可扫描性
用户后来明确希望输出结果在对话模式中也保留高亮能力，这个判断非常准确。

因为 dialog transcript 一旦失去高亮，就会退化成大段纯文本，用户很难快速识别：

- 报错
- 成功状态
- 文件类型
- 命令输出结构

所以高亮不仅是“更好看”，更是 transcript 结构化呈现的一部分。

## 当前 UI 最值得保留的设计哲学

### 1. 界面必须服务于运行时语义
不是为了好看加模式，而是因为模式真的不同：

- 普通命令
- 交互式命令
- AI CLI 工作流

界面只是把这个事实表达出来。

### 2. 极简不等于没有层次
即使最终追求纯白背景、黑字、无圆角、紧凑拼接，也依然需要保留：

- pane 边界
- 激活态
- 模式提示
- 分屏关系

真正的极简，是保留最少但足够强的界面信号。

### 3. 长时间使用比首次惊艳更重要
终端不是海报，不是 landing page，也不是一次性演示。

它是会被连续使用几小时的工具。

因此一切 UI 决策都应优先考虑：

- 低疲劳
- 高对比
- 明确状态
- 快速扫描
- 不抢内容注意力

## 后续最值得继续做的三件事

### 1. 继续完善主题 token 化
把更多视觉属性收敛到统一 token：

- pane 边界
- 输入框边框
- AI mode 强调
- transcript 分隔线
- 选中态

这样后面加主题不会把样式写散。

### 2. 让设置页真正成为视觉控制中心
设置页后续最值得支持：

- 主题选择
- 自定义主色
- AI mode 强调色
- 字体族
- 字号和行高

这会直接提升产品完成度。

### 3. 统一 dialog transcript 的内容级高亮策略
继续把输出中的：

- ANSI 颜色
- 错误块
- 命令头
- 分隔线
- 路径和关键字

做成稳定的内容层呈现规则，而不是零散补丁。

## 一句话总结
PRAW 当前 UI 设计最有价值的地方，在于它不是单纯追求“像某个产品”，而是在把终端、分屏工作区、AI 工作流和可定制视觉系统逐步融合成一套统一界面语言。

它的真正亮点是：状态被看见，内容被尊重，主题服务工作流，而不是反过来。
