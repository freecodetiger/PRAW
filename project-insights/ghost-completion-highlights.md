# PRAW 幽灵补全技术亮点

## 结论
PRAW 的幽灵补全不是把一个 AI API 生硬塞进输入框，而是把“终端预测”拆成了三个本质层次：

- 本地即时匹配
- 可解释的上下文收集
- 可替换的远端智能预测

这套设计的关键价值不在于“能补全”，而在于它保持了终端场景最重要的三个属性：

- 响应要快
- 结果要像命令而不是像聊天
- 失败时不能破坏基础输入体验

## 这个问题的本质
终端补全和普通文本补全不一样。用户不是在写自然语言，而是在输入：

- 命令
- 参数
- 路径
- 工具链惯用语
- 个人工作流中的高频短语

所以真正的问题不是“如何生成一句更聪明的话”，而是：

- 如何在极短延迟下预测下一条可执行命令
- 如何避免补全污染用户输入
- 如何让预测结果和当前机器、当前目录、当前项目有关

如果这三个问题不拆开，补全系统很容易变成一个看起来智能、实际不可依赖的装饰层。

## 当前方案为什么合理

### 1. 先把补全分层，而不是一上来全交给模型
PRAW 当前的补全设计天然适合分层：

- 常用语补全负责零延迟命中
- 本地前缀匹配负责稳定可预期
- AI 补全负责在需要时补足“猜测能力”

这比单一 LLM 驱动方案更稳，因为终端里最常见的高频输入其实并不需要每次都访问云端。

例如：

- `cd projects/`
- `claude`
- `codex`
- `git checkout`
- `docker logs`

这些命令的预测价值，很多时候来自用户习惯和当前前缀，而不是来自一次昂贵的生成。

### 2. 只在 dialog 模式承载幽灵补全，降低冲突面
这个项目已经验证过一件事：classic 模式和终端原生输入行为耦合非常深。

经典终端里存在：

- PTY 回显
- shell 补全
- ANSI 输出
- 输入法组合态
- 交互式程序接管 stdin

如果在这层硬塞一套前端幽灵补全，很容易引入白屏、卡顿、回显冲突或输入状态错乱。

因此当前把幽灵补全聚焦在 dialog 模式，本质上是在做边界管理：

- dialog 模式负责“增强输入体验”
- classic 模式负责“完全兼容原生命令行”

这是一种克制但正确的决策。

### 3. 常用语补全先于 AI 补全，是对终端场景的尊重
终端中最值得优先优化的，不是最炫的补全，而是最常用的补全。

当前方案里，常用语可以：

- 持久化
- 从设置导入
- 基于公共前缀匹配
- 在多候选下用快捷键切换
- 用 `Tab` 明确接受

这意味着补全不是“模型替用户做主”，而是“系统把最可能的惯用操作递给用户”。

这种交互非常接近成熟终端用户的心智模型。

## 为什么它具备继续升级成 AI 预测补全的基础

### 1. 上下文接口天然明确
你前面定义的 AI 预测补全输入结构，本质上是对“终端上下文”做结构化表达：

- 当前前缀
- 当前目录
- 历史命令
- Git 分支
- 会话标识

这很重要，因为它避免了一个常见错误：
不是把整个终端输出原样扔给模型，而是只传真正影响预测的上下文。

这种接口设计有三个直接好处：

- 易于脱敏
- 易于缓存
- 易于替换模型供应商

### 2. OpenAI 格式兼容层让模型接入被抽象掉
项目已经在 AI 能力侧形成一个重要方向：
尽量统一为 OpenAI 兼容请求格式。

这意味着幽灵补全未来不需要和每个模型厂商深度绑定。它只需要：

- provider 配置
- model 名称
- 一套统一请求协议

这样做的意义是架构级的：

- 前端补全逻辑不需要知道具体厂商差异
- Rust 或服务网关可以统一做超时、限流、错误映射
- 后续从 GLM 扩展到其他供应商时不会重写补全 UI

### 3. “至少输入 2 个字符再预测”是高性价比阈值
终端输入的前缀信息密度很高，但 1 个字符通常噪声太大。

把触发阈值设为至少 2 个字符，本质上是在平衡：

- 请求成本
- 预测质量
- UI 稳定性

例如：

- `c` 太模糊
- `cd` 已经明显进入目录切换语境
- `gi` 已经能高度聚焦到 Git
- `ku` 已经可能指向 `kubectl`

这个阈值不是拍脑袋，而是对命令行前缀信息量的工程化取舍。

## 当前方案最有价值的工程智慧

### 1. 补全只是建议，不直接改写输入
无论是常用语补全还是未来的 AI 预测补全，核心都应该是：

- 用户看到建议
- 用户主动接受
- 未接受时输入不被篡改

这是终端体验的生命线。

因为一旦系统在用户输入流里“自作主张”，终端就从工具变成了干扰源。

### 2. 候选切换比一次性唯一答案更适合终端
普通聊天产品喜欢只给一个答案，但终端补全不一样。

对于：

- `git c`
- `docker l`
- `kubectl g`

这类前缀，往往存在多个合理候选。

因此支持候选列表和快捷键切换，技术上代表一种更成熟的判断：
补全系统输出的不是“真理”，而是“排序后的可能性集合”。

### 3. 本地文件结构与系统信息纳入上下文，才是真正的“终端智能”
终端不是孤立输入框，它和当前机器强相关。

如果未来补全能安全利用：

- 当前目录结构
- 是否处于 Git 仓库
- 常见子目录名称
- 当前 shell / OS
- 常用工具是否存在

那么预测就会从“通用命令生成”变成“当前环境下的高概率建议”。

这才是真正符合终端产品价值的智能增强。

## 风险控制为什么关键
幽灵补全在终端场景里有几个天然风险：

- 误补全高危命令
- 上传敏感路径或密钥
- 高频请求导致限流或卡顿
- 输入法状态与前端预测冲突

你已经在方案里给出了正确方向：

- 路径脱敏
- 高危命令过滤
- 本地缓存优先
- 超时后回退到本地补全

这说明补全系统被当作“生产力能力”而不是“演示能力”设计，这一点非常重要。

## 后续最值得继续做的三件事

### 1. 做命令族优先级模型
优先围绕真实高频命令族优化：

- `cd`
- `git`
- `docker`
- `ssh`
- `ls/cat/less/tail`
- `systemctl`
- `kubectl`

这样最能直接提升体感。

### 2. 做本地缓存与目录维度缓存
相同目录下的常见前缀，命中率会非常高。

把缓存 key 设计成：

- 用户
- 前缀
- 目录

会让补全的“像你自己”程度明显增强。

### 3. 做供应商无关的补全网关
补全是一个长期演进能力，不能绑死在某个模型厂商或某个模型名上。

因此最优方向依然是：

- 前端只认统一协议
- 后端或网关统一转发
- provider 可替换

## 一句话总结
PRAW 的幽灵补全亮点，不是“接了 AI”，而是它把终端预测拆成了一个可演进、可降级、可持久化、可控风险的系统。

换句话说，这不是一个会说话的输入框，而是一套围绕真实终端工作流构建的预测基础设施。
